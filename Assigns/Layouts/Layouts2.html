Introductory Android

Overview

This and later exercises will draw on the introductory courses supplied at

https://developer.android.com/courses

and for the first homeworks, this course in particular:

https://developer.android.com/courses/android-basics-kotlin/course

Each of these courses is somewhat "code by the numbers" overview of Android dev topics. To use them effectively, note:

  • The videos are mostly just cute rah-rah, so you can skip those.

  • The codelab walkthroughs are where the serious content is. For these, actually write the code, don't just copy-paste it. And:

    • First, read through a portion of the presentation, up to an entire path depending on how good your memory is.

    • Then, *set aside the example code and presentation*, and write the code yourself, from memory as much as possible, checking the presentation only as absolutely necessary.

    • As you write each line, be sure you understand what it does and why.

  • A canned example like these codelabs is just a starting point.

    • Do deeper exploration of the classes used, via the API reference and other sources. Find out how they're organized, what their inheritance relationships are, etc. Bookmark useful points of the API reference.

    • Based on what you learned, explore "around" the code, trying different alternatives to see how they work.

    • I'll help with this by assigning required homework explorations surrounding the code in the walkthroughs, but learn also to do this for yourself.

Unit 2: Layouts: Create XML Layouts for Android Pt 2

Much of this path centers on what is sometimes called "paint" -- the artistic and aesthetic tuning of a UI's colors, fonts, icons, etc. It's easy to dismiss this as window dressing, but it's not. People like to use good-looking UIs, and industrial design and aesthetics are an interesting area, with significant technical elements.

  1. For the first step, use the concepts of theme and style to set up a version of TipTime that has Principia blue and gold coloring for its primary and secondary colors. And, arrange the code so that you can, by changing just one word in the source, switch the app (after rebuilding and restarting) between Prin colors and the original purple-ish scheme. Unsurprisingly, this will involve some independent research on how the magic of themes works, and in particular how your layout "knows" what theme to apply.

  2. Build for a recent Android version and be sure you can also switch, while the app is running, between day and night themes, with appropriate shifts in coloring, in both original and Prin themes. Keep an older version around as well.

  3. After #1 and #2 are working, so some research on styles vs themes, and be sure you understand both. Note for instance, that the style of the main button in TipTime is Widget.MaterialComponents.Button. Consider how this button "knows" to use the primary color as its background. Using this, arrange code so that the button instead uses the theme secondary color for its background, and so that switching themes automatically, without altering the button configuration, causes the button to show whichever secondary color is appropriate for the theme.

  4. If you accomplished #3 by altering one of the color properties of the button, that's fine. But now, if you didn't already, leave the color properties as they originally were, and instead alter *only* the style of the button, giving it a new style that is exactly like Widget.MaterialComponents.Button in all respects except that it uses the secondary color for background. (Note that this is how the Widget.MaterialComponents.Button style itself ties theme elements to specific aspects of the widget.)

  5. If you wanted to set a different theme for two different layouts in an application (e.g. in two different activities) is this allowed, and how?

  6. "Mipmapping" is a much larger concept outside of Android, beyond the specific application in the path. Look it up, and briefly describe how it's used in other areas. For instance, how might mipmapping be useful even if the object in question is not changing its physical size? Why is mipmapping useful, as opposed to just having one image and computationally scaling it? Can you do mipmaps with just a single image file? How does this work?

  7. Which of the two major ways Android represents icons is a form of SVG? Why? Which of the two is your Android device (or emulator) using? Why?

  8. After doing the path exercise to change the tiptime icon to a white bell on green background, redo it as a gold bell on a shaded blue background, and properly regenerate all assets needed. Do this by selected hand-modification of the XML assets for the white-on-green; it doesn't take much. Be sure this works both on older Android versions (pre-26) and newer ones.

  9. After doing the switch to Material for the EditText and Switch, do it also for the Button, replacing it with a material "floating action button" (pick the right type of floating action button based on the docs). Maintain the customized style, but be sure you derive from the right base style now that you've changed the button.

  10. When adding icons, use a dollar-sign icon for the Cost of Service label. Be sure the 3-line editText we added is also indented like the other components.

  11. Mess around a bit with style and theme so that

    • the dollar-sign icon is always Principia blue, night or day

    • the service icon is unchanged from the original XML, and yet it shows prin blue during daylight, and its default behavior during night.

    • The round-up arrow is whatever secondary color is specified by the theme, and thus changes appropriately night and day.

  1. When setting up the key handler for removing the entry keypad, answer the following:

    • If you return "true" from the handler, regardless of keyCode, what is the effect on the UI? Why?

    • Look up the type of the parameter for setOnKeyListener. This is not a function, but we're passing a lambda. How does that work, and what three letter acronym describes what the parameter is, and why the lambda matches it?

    • Why do we have to use a lambda in setOnKeyListener? The handleKeyEvent function is a function -- won't it suffice directly, e.g. setOnKeyListener(handleKeyEvent)? What if we added a third parameter to handleKeyEvent? What then?
